## 0. 学習の原則

1. **観測で学び、質問で学ばない。** アンケート・NPS・感想の督促は送らない。
   聞いた瞬間に「テストされている場」になり、静けさが壊れる。
   ユーザーが自発的にDMで語ったことだけが、聞いてよい声。
2. **行動は言葉より正直。** 「毎日とっているか」は、どんな感想よりも強いシグナル。
3. **30人は統計でなく、顔の見える30人。** 割合でなく人数で数える。
   「継続率67%」でなく「20人つづいている、10人止まった」で考える。
4. **仮説を先に書く。** 数字を見てから物語を作らない（後知恵の禁止）。§2に事前予想を固定した。

## 1. 測るもの（すべて既存データから。追加計測の実装はしない）

### 北極星
- **初週継続**: オンボ完了から7日のうち5日以上「とった」人の数

### ファネル（人数で追う）
| 段 | 定義 | データ源 |

### リズム（週次で眺めるだけ）
- 曜日別の「とる」人数（谷曜日の把握。平日/週末の型）
- とる時刻の分布（夜集中か、日中分散か——「昼に撮って夜受け取る」設計の実態確認）
- ひらく率: 届いた通のうち当日開封された割合
- 20時直後（20:00〜21:00）のひらく集中度（儀式が機能しているかの代理指標）

### 質のシグナル（数えるだけ・介入しない）
- 自発的DM（感想・写真の話・不具合報告）の件数と内容の型
- reject件数と型（MODERATION-CANONグレーログと同一）
- photo_reports件数（0が続くことも情報）

## 2. 事前予想（後知恵防止のため固定。7月末に答え合わせ）

- F1→F2の脱落が最大の谷（PWAインストールの摩擦）: 応募の2〜3割はオンボ未完で止まる
- 初週継続は30人中 **12〜18人**（40〜60%）
- 谷曜日は月曜（週末の写真ストックが切れる日）
- 自発的DMは初週に集中し、二週目にゼロ近くまで減る（減ることは失敗ではない。
  静かに使われている状態こそ設計意図）

## 3. 週次レポートの型（毎週日曜夜・30分で書ける形）

```
# 週次 YYYY-MM-DD
## 人数
F1応募: N (+週増分) / F2完了: N / F4初週継続: N / F5二週目: N
## 今週の一枚の絵
（曜日別とる人数の推移を3行で。数字の羅列でなく「何が起きたか」を一言）
## 予想との差分
（§2のどれが当たり、どれが外れたか。外れの理由の仮説1つまで）
## 鏡:
（参照したい他サービス・他アカウントがあれば1行。数字のベンチマークにせず、距離感を見る）
## 声（あれば）
（自発的DMの型を1〜3行。原文引用は本人の許可なくどこにも書かない）
## 来週かえること・かえないこと
（かえるは1つまで。かえない宣言も明記——静けさの運用は「動かない判断」の記録が資産）
```

- レポートは `docs/weekly/YYYY-MM-DD.md` にコミット（Claudeとの週次レビューの入力になる）
- **かえるは週1つまで**。30人の観測環境で同時に2つ変えると、何が効いたか永久に分からない

## 4. 介入のルール（観測を汚さないために）

- 不具合・つまずきへの対応: 即やる(観測より体験)
- コピー・デザインの微修正: 週1つまで（§3）
- 新機能の追加: 先行期はしない（P3後の判断材料を濁らせない）
- 個別ユーザーへの「使ってますか?」連絡: **禁止**。7日以上止まった人も追わない。
  止まった理由はP3後にオンボ改善で構造的に対処する（個人を追うのは監視であり、静けさの反対）
