> 関連: `03142d5 Clarify delivered photo deletion behavior` / 規約A-3(3) / docs/legal-docs-inventory-2026-07.md

## 0. 目的と原則

アカウント削除（退会）時、**配達済みのねがおは受け取った側の記録として画像ごと残す**。

原則:
- ねこだよりは手紙。一度届いたものは、差出人が去っても受取人の手元に残る
- 消すのは「個人に紐づくもの」（認証情報・メール・アカウント行・未配達データ）
- 残すのは「相手の記録になったもの」（配達済みdelivery行と、その参照するStorageオブジェクト）
- 残った写真から削除済みアカウントへ逆引きできないこと（匿名性の切断を検証で保証）

## 1. 変更対象

### 1.1 `deleteAccountStoredData()`（および削除runbook）

現状: アカウント削除でStorageオブジェクトを削除するため、配達済み写真が受信側で表示不能になる可能性がある。

変更後のStorage削除ロジック:

```
対象ユーザーの全Storageオブジェクトについて:
  if cat_moment_deliveries に当該オブジェクトを参照する行が存在（status=delivered）:
    → 残す（削除しない）
  else:
    → 削除する（未配達・下書き・プロフィール画像等は従来通り消す）
```

- 参照判定は `cat_moment_deliveries.photo_url`（または実装上の参照カラム）とStorageパスの突合で行う。
  署名付きURL文字列でなく、正規化したオブジェクトパスで比較すること
- `cat_moments` 行の扱い: 配達済み参照を持つmomentは、
  (a) 行を残して所有者カラムを匿名化する か
  (b) 行を消してdeliveryを自立させる（deliveryが表示に必要な情報を自己完結で持つ場合のみ可）
  のどちらかを実装調査の上で選択。**受信側の表示が cat_moments 行に依存しているなら (a) 一択**。
  選択と根拠を実装報告に含めること
- 削除runbook（手動手順）も同じ分岐に更新

### 1.2 匿名性の切断（残す側の検証）

アカウント削除後、残ったdelivery行・moment行・Storageオブジェクトのいずれからも、
削除されたユーザーの認証情報（メール・auth user id）へ到達できないことを確認する。

- auth.users 行削除後に FK・カラム経由の逆引きが残らないこと
- 匿名ID（anonymous_id）は残ってよい（それ自体は個人に到達しない前提。
  もし anonymous_id → メール等への対応表が存在するなら、その対応行の削除を本削除フローに含める）

### 1.3 削除確認ダイアログへの一文追加

アカウント削除の確認UIに、実行前の説明として一文を追加する:

> すでにお届けしたねがおは、受け取った方の記録に残ります。

- 位置: 削除の最終確認ボタンより前、視認できる位置
- トーン: 警告色にしない。事実の静かな告知として本文と同じ調子で置く
- 文言は仮。**最終文言は中西さんが確定**（実装はプレースホルダ差し替え可能な形に）

### 1.4 削除案内（/account-deletion）の文言簡素化

`03142d5` で分岐させた「写真1枚削除」と「アカウント全体削除」の説明を、一本に戻す:

> **すでにどなたかに届いた写真について**
> 一度とどいた ねこだよりは、受け取った方の記録として残ります。
> 写真を削除しても、アカウントを削除しても、それ以降あたらしく届くことはありません。
> もし、どうしても届いた先からも消したい事情がある場合は、お問い合わせください。
> 事情をうかがったうえで対応を検討します。

- 「届いた先からも消したい事情」の窓口を残すのは、権利侵害系の例外対応（規約A-3(4)）の受け皿

### 1.5 legal-docs-inventory の更新

「アカウント削除はStorage削除により表示不可の可能性あり」の追記を、
本仕様の反映後は「配達済みは保全される仕様に変更済み（本spec参照）」に更新。

## 2. スコープ外

- 受信側の画像読み込み失敗時のグレースフルなフォールバック表示
  → 別タスク（P2）。本仕様が入っても、孤児31件・Storage事故への保険として価値が残るため
  spec化は別途。本仕様で新規の欠けは発生しなくなる
- 未参照Storageオブジェクトの定期GC → 既存のP2タスク箱（grant-hardening等と同枠）
- 写真1枚削除のロジック → 変更なし（現状で規約と整合済み）

## 3. テスト

### 3.1 E2E（自動）

1. ユーザーAが投稿 → 承認 → ユーザーBに配達 → Aがアカウント削除
   → Bの受信画面で当該ねがおが**画像ごと表示され続ける**
2. ユーザーAが投稿（未配達のまま）→ Aがアカウント削除
   → 当該Storageオブジェクトが削除されている
3. アカウント削除後、Aの認証情報で到達できるデータが残っていない
4. 削除確認ダイアログに1.3の一文が表示される

### 3.2 本番確認（読み取り＋テストアカウントのみ）

- テストアカウントで 3.1-1 の一巡を本番で実施（実データは使わない）
- 既存ユーザーデータへの遡及処理は**行わない**（本仕様は今後の削除に適用）
