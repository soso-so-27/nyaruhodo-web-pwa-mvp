# Analytics Runbook

更新日: 2026-07-18

## 目的

公開後の状態を、推測ではなく管理画面の構造化データから同じ基準で判断する。分析は読み取り専用とし、修正や本番操作は別の明示指示で行う。

## 合言葉

| 合言葉 | 対象 |
|---|---|
| `初動分析` | 公開側の直近60分と公開後累計 |
| `夜便分析` | 公開側の直近60分と当日。20時便の予約、確認、成立、開封 |
| `週次分析` | 公開側の7日。流入、写真保存、再利用、端末、未解決 |
| `分析` | 19時前は初動分析、19時以降は夜便分析 |

## 事実の扱い

- 数字は人やアカウントではなく、ログインIDまたは匿名ブラウザIDとして報告する。
- `direct` は「流入元不明」とし、ThreadsやDM由来と推定しない。
- 管理画面の「公開側」を使用し、管理者・既知のQAを除外する。ただし過去の匿名QAは混ざる可能性を注記する。
- 同一ID・同一原因の連続イベントは、管理画面の論理的な「出来事」単位で数える。
- 後続で到着、開封、成功まで進んだ失敗は回復済みとし、未解決に含めない。
- 母数が小さい間は率だけで良否を断定せず、分子と分母を併記する。
- 前回スナップショットがない項目は「増減不明」とする。

## 運用判定

- **赤**: 直近30分の未解決、または60分以内に同じ失敗が2 ID以上へ波及。対象IDの後続イベントを直ちに確認する。
- **黄**: 30分より前の未解決のみ。新規再発を監視する。
- **緑**: 未解決なし。回復済み・想定内は別記する。

20時15分以降は、`20時便の確認を開始` と `20時便が成立` の識別ID差を最優先で確認する。差があり、未解決の夜便確認失敗がある場合は赤として扱う。

## 報告書式

```text
結論: 緑 / 黄 / 赤

事実
- 入口: 現在値（前回差）
- 写真保存: 現在値（前回差）
- 初回到着・開封: 現在値（前回差）
- 未解決: 出来事件数 / 影響ID / 最新時刻
- 20時便: 予約 / 確認開始 / 成立 / 開封

判断
- 事実から言えること
- まだ言えないこと

次にすること
1. 最大3件
```

## スナップショット

- 保存先: `artifacts/analytics-snapshots/`
- Gitには追加しない。
- 管理画面の「Codex分析用データ」だけを保存し、生イベント一覧や認証情報は保存しない。
- 比較は同じ `audience` と `period` の直前スナップショットに対して行う。

## Codexの取得経路

- 通常は本番管理画面へのブラウザログインを使わず、`scripts/read-production-analytics.mjs` で本番DBを読み取り専用集計する。
- リポジトリにリンク済みの本番SupabaseへCLI認証で接続し、接続情報はプロセス内だけで使う。リポジトリや一時ファイルへ保存しない。
- スクリプトはローカルSupabaseを本番として読むことを既定で拒否する。
- DB集計が使えない場合のみ、管理画面の構造化JSONを代替経路として使う。
