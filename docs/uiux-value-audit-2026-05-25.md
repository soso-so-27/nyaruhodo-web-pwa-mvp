# UI/UX価値検証メモ

Last updated: 2026-05-25

## 目的

にゃるほどの全ページが、ユーザーにとって分かりやすい価値を持っているかを検証する。

今回の検証では、単なる利用手順書ではなく、以下を組み合わせた。

- 画面ごとの役割チェック
- ユーザー手順ごとのウォークスルー
- モバイルUIの基本原則に沿ったスコアリング
- マネタイズに向けた価値のつながり確認
- 不十分な導線、重複、認知負荷の洗い出し

## 前提にするプロダクトループ

にゃるほどの基本ループは以下。

1. 猫を見つける
2. アプリを開く
3. みっけ / おせわ / 写真を1つ残す
4. トリセツやコレクションに価値がたまる
5. 残しておきたい気持ちが生まれる
6. アカウント同期や将来の有料保存につながる

このループに貢献しない要素は、基本的に削るか、別ページへ移す。

## 評価方法

### 1. 役割適合

`docs/app-role-map.md` を基準にした。

- ホーム: 猫を見た瞬間に記録する場所
- トリセツ: たまっていくナレッジ棚
- コレクション: 写真を集めたくなる棚
- ねこ: プロフィールと写真管理
- 設定: アカウント、同期、データ、ルール
- 診断オンボーディング: 最初の猫作成と最初のトリセツ作成

### 2. 主要手順ウォークスルー

新規ユーザーと既存ユーザーの行動を想定して確認した。

- SNSなどから入る -> 診断 -> ホーム
- ホームでみっけ / おせわ / 写真を残す
- トリセツで分かったことを見る
- コレクションで写真をためる
- ねこタブで写真やプロフィールを整える
- Googleログイン後、設定から同期 / 復元する

### 3. ヒューリスティック評価

以下の観点で採点した。

- いま何をすればよいか分かるか
- 画面の役割が一目で分かるか
- 読まないと分からない要素が多すぎないか
- 主要アクションが押しやすいか
- タブ、ボトムシート、カードの挙動が一貫しているか
- 失敗や不安を防げているか
- 継続利用したくなるか
- 将来の課金理由につながっているか

参考にした考え方:

- Apple HIG: タブは安定した主要目的地であるべき
- Material Design: ボトムシートは一時的な集中タスクに向く
- Nielsenのユーザビリティ原則: 状態の可視化、現実の言葉、記憶より認識、最小限の情報量

## スコア基準

## `/home`

### 役割

猫を見た瞬間に開いて、1つだけ残す場所。

### 評価

| 観点 | スコア |
| --- | ---: |
| 感情的な引力 | 4 |
| 主アクションの明確さ | 4 |
| 片手操作 | 4 |
| おすすめカードの意味 | 3 |
| 視線誘導 | 4 |
| 習慣化 | 4 |

総合: **4.0 / 5**

### 良い点

- 写真背景にUIをのせる方向は強い。
- 猫を見た瞬間に「みっけ」しやすい。
- ボトムシートで入力する設計は、短い記録に向いている。

### 課題

- 「あなたへのおすすめ」が、アクション、発見、通知を全部背負いやすい。
- おすすめカードとトリセツの役割が重複しやすい。
- カードが多いと、ホームが読む画面になってしまう。

### 改善案

おすすめカードを3種類に分ける。

1. アクション
   - みっけ
   - おせわ
   - 写真

2. 価値提供
   - 発見
   - 最近の変化

3. システム
   - 同期
   - お知らせ

ルール:

- アクションカードはボトムシートを開く。
- 価値提供カードはトリセツや詳細へ送る。
- システムカードは設定やアカウントへ送る。

ホームは「読ませる」より「押せる」を優先する。

### 評価

| 観点 | スコア |
| --- | ---: |
| 役割の明確さ | 4 |
| 情報構造 | 3 |
| 認知負荷 | 3 |
| 長期価値 | 4 |
| ホームとの整合性 | 3 |

総合: **3.4 / 5**

### 良い点

- 「診断結果」と「みっけから分かったこと」を分ける方向は正しい。
- 未開封の深掘り診断を置く構造は、将来の継続利用と相性が良い。

### 課題

- ユーザーにとって、以下の違いがまだ一目では分かりにくい。
  - みっけから分かったこと
  - 診断結果
  - 未開封の診断
- 件数や問数は、ユーザー価値が弱い。
- 表現が多いと、ナレッジ棚ではなくダッシュボードに見える。

### 改善案

最小構造はこれでよい。

1. 発見
   - みっけから分かったこと
   - 変わっていく知識

2. 診断結果
   - 回答済みの診断結果
   - 開くと詳細を見る

3. 未開封
   - これから答えられる深掘り診断
   - ロック / サンプル診断

避けたい言葉:

- みっけから
- これから開く
- 件数を主役にする表現

使いやすい言葉:

- 発見
- 診断結果
- 未開封
- タイプ診断
- ごきげん診断
- ごはん診断

## 利用手順ベースの検証

### 手順A: 初回ユーザー

1. `/` を開く
2. 診断を始める
3. 名前、写真、基本情報を入れる
4. 診断に答える
5. 結果を見る
6. ホームへ進む
7. みっけを1回する
8. トリセツに結果が残る

合格条件:

- 診断結果が「最初のトリセツ」だと分かる
- ホームが記録する場所だと分かる
- アカウント作成を強制されない

現状の懸念:

- 診断結果からホームへ進む意味を、もう少し強くしてよい。

### 手順B: 日常利用

1. 猫を見る
2. ホームを開く
3. みっけ / おせわ / 写真から1つ残す
4. 閉じる

合格条件:

- 読まずに押せる
- 1つだけ残せばよいと分かる
- ロックやカウントダウンが分かる

現状の懸念:

- おすすめカードが多いと、行動前に読ませてしまう。

### 手順C: 価値確認

1. トリセツを開く
2. 発見を見る
3. 診断結果を見る
4. 未開封診断を見る

合格条件:

- みっけから分かったことと診断結果の違いが分かる
- これから増える余地が分かる

現状の懸念:

- セクション名とカード密度をさらに削りたい。

### 手順D: 写真を集める

1. コレクションを開く
2. 今日の見つけたい姿を見る
3. 写真を追加する
4. 「見つけた」状態を見る
5. 必要ならshareする

合格条件:

- 集めたくなる
- 写真追加後に達成感がある

現状の懸念:

- 達成演出がまだ控えめ。

### 手順E: アカウント同期

1. Googleログイン
2. ホームへ戻る
3. 設定を開く
4. この端末のデータを保存する
5. 別端末で復元する

合格条件:

- データが消えない
- 同期と復元の違いが分かる

現状の懸念:

- 文言はもう一段やさしくできる。

## 改善優先度

## P0: 小さなSNSβ前にやりたい

1. トリセツの情報構造をさらに削る
   - 発見
   - 診断結果
   - 未開封

2. ホームおすすめカードの種類を固定する
   - アクション
   - 価値提供
   - システム

3. コレクションの写真追加後の達成感を強める
   - 〇〇を見つけた
   - share

4. 問い合わせ先を決める
   - privacy / terms / beta用

## P1: β中に見る

1. 診断開始 -> 完了率
2. ホーム閲覧 -> みっけ率
3. トリセツ閲覧率
4. コレクション写真追加率
5. 同期成功率

数値が悪いところからUIを削る。

## P2: 課金前

1. 無料 / 有料の境界をUIに出す
2. 長期保存の価値を説明する
3. 設定にプラン表示を作る
4. privacy / termsを法務確認する

## 次に実装するなら

おすすめ順:

1. トリセツの情報構造をさらに削る
2. ホームおすすめカードの分類を固定する
3. コレクションの「見つけた」後の達成感を強める
4. 設定の同期 / 復元文言をやさしくする
5. 問い合わせ先を決めて、privacy / termsへ反映する
