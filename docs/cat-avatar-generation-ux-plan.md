# うちの子アバター生成UX設計

## 1. 目的

猫カードに「自分の猫がいる感じ」を出す。

現在:

- 状態連動アイコン
- 毛色アクセント
- neutral fallback

次の狙い:

- 写真から、この子専用のアバターを作る
- アプリ内の猫カードに反映する
- 将来的に、状態別ポーズや週次まとめにも展開できるようにする

## 2. 基本方針

- いきなり写真アップロードを初回オンボーディングで要求しない
- /cats の「この子のこと」から任意で設定できるようにする
- 写真はユーザーが明確に選んだものだけ使う
- 生成結果が気に入らない場合のやり直し導線を用意する
- 生成画像は「診断精度」ではなく「自分の猫専用感」のために使う
- 医療判断や猫種判定には使わない

## 3. 入口

入口は /cats に置く。

候補文言:

- この子のアバターを作る
- 写真からアバターを作る
- ミケらしいアイコンを作る

避ける文言:

- AIで猫を判定
- 猫種を推定
- 正確なアバターを作る
- 本物そっくりにする

## 4. UXフロー

### Step 1：/cats から開始

表示例:

この子のアバター  
写真から、アプリ用の小さなアイコンを作れます。

[写真を選ぶ]

### Step 2：写真選択

注意文言:

猫の顔や全身が分かりやすい写真を選んでください。  
あとで変更できます。

### Step 3：生成前確認

表示例:

この写真から、ミケのアバターを作ります。

[作成する]  
[写真を選び直す]

### Step 4：生成中

表示例:

ミケのアバターを作っています...

注意:

- 長く見せすぎない
- 失敗時の戻り先を用意する

### Step 5：生成結果表示

表示例:

ミケのアバターができました

[これにする]  
[もう一度作る]  
[写真を選び直す]  
[やめる]

### Step 6：保存

保存後:

- /cats の猫カード
- /home の猫カード

に反映する。

## 5. 生成画像の使い方

まず使う場所:

- /home の猫カードのアバター
- /cats のいま見ている子
- 猫一覧チップの小さなアイコン

まだ使わない場所:

- 診断結果
- 病院用レポート
- 週次まとめ
- ステッカー
- 状態別アニメーション

## 6. 既存の状態連動アイコンとの関係

現状:

- いま/さいきんの signal に応じて、sleeping.png などの状態アイコンを表示している

アバター生成後の方針候補:

### 案A：生成アバターを常に優先

メリット:

- 自分の猫感が強い

デメリット:

- 状態連動が弱くなる

### 案B：生成アバターをベース、状態は小アイコンで補足

例:

- 生成アバターを表示
- 右下に sleeping / playing などの小バッジを表示

メリット:

- 自分の猫感と状態感を両立できる

デメリット:

- UI調整が必要

### 案C：状態連動アイコンを維持し、生成アバターは /cats だけ

メリット:

- 実装が軽い

デメリット:

- ホームの専用感が弱い

推奨:

案B。  
生成アバターをメインにし、状態は小さなバッジ/アクセントで表現する。

## 7. 保存先の考え方

MVP localStorage段階:

- 生成画像URLまたはbase64はlocalStorage保存に向かない
- まずは実装しない
- 本実装ではStorageが必要

将来:

- Supabase Storage などに画像保存
- cats.avatar_url に保存
- 生成元写真と生成済みアバターは分けて扱う

注意:

- 現時点ではDB変更しない
- 実装時にはStorage/認証/DB化との関係を再確認する

## 8. プライバシーと注意文言

必要な説明:

- 選んだ写真はアバター作成のために使います
- 医療判断や猫種判定には使いません
- 生成結果はあとで変更できます
- 気に入らなければ使わなくても大丈夫です

避けること:

- 写真を勝手に送信する
- 生成結果を強制採用する
- 写真を削除できない状態にする
