# Instagram 投稿3 公開後ライブ運用シート

確定日: 2026-07-17
運用更新日: 2026-07-21（Phase 1）
対象期間: 投稿直後から翌朝まで
起点: `docs/instagram-post3-final-2026-07.md`
判断の詳細: `docs/OPERATIONS-PLAYBOOK.md` / `docs/analytics-kpi-inventory.md`

この文書は、公開後に迷わず動くための一枚である。通常の反応は観測し、
写真・配達・プライバシー・課金に関わる異常だけを即時対応する。

## 0. 今日の原則

1. **写真と配達の信頼を最優先**する。見た目や文言の違和感では公開を止めない。
2. **同じ操作を何度も試してもらわない**。重複保存や二重決済を避けるため、証跡を先に取る。
3. **19:00〜21:00はデプロイしない**。P0対応だけを例外とする。
4. **率より人数を見る**。入口が5人未満の間は一人ずつの経路を見る。
5. **質問で観測を作らない**。「使ってみてどうでしたか」は送らず、自発的な声だけを受け取る。
6. **公開日に新機能を足さない**。P1以下は記録し、週次で一つだけ変える。

## 3. 赤の意味を分ける

| 止まった場所 | 第一解釈 | 扱い |
| --- | --- | --- |
| 入口を見たが写真選択を押さない | 興味・タイミングの差 | 観測のみ |
| 写真選択を押したが保存されない | 写真読込または保存の問題 | 1人目で調査、同型2人でP0 |
| 写真保存済みだが即時便が来ない | 製品の中核障害 | 1件でP0 |
| 即時便は来たが開かない | 操作や関心の差 | 操作不能でなければ観測のみ |
| 初回写真保存済みだが次回20時便が予約されない | 製品の中核障害 | 実際の未予約を確認したら1件でP0 |
| 既存の予約対象が初回写真で上書きされた | 予約の整合性障害 | 1件でP0 |
| 予約済みだが次の20時以降に再訪しない | 未再訪 | 障害ではない |
| 20時便確認を開始したが成立しない | 配達障害 | 再試行後も失敗ならP0 |
| 成立済みだが封筒を開かない | 未開封 | 障害ではない |

分析イベントは端末側キューから遅れて届くことがある。人数の差だけでP0と断定せず、
本人の画面、DM、管理画面の実データ、delivery行のいずれかで実際の不成立を確認する。

## 4. 緊急度と初動

### P0: 入口を止める

次のどれか一つでP0とする。

- 他人の写真・名前・場所など、見えてはいけない情報が見えた
- 「自分だけ」の写真が配達候補または他者表示になった疑いがある
- 写真の消失、誤配、重複配達が確認された
- 写真保存済みなのに即時便が成立しないことを、画面または実データで確認した
- 初回写真の保存後に次の20時便が予約されないことを、画面または実データで確認した
- 同じ予約日にあった既存の対象写真が、初回写真によって上書きされた
- 20時便の確認開始後、再試行しても成立しないことを、画面または実データで確認した
- 同じ写真読込・保存停止が別の2人に起きた
- 白画面、戻れない画面、操作不能から復帰できない
- 二重請求、金額違い、決済成功後の契約不整合がある

初動は順番を変えない。

1. 時刻、端末、開いた場所、止まった画面を記録する。
2. 同じ操作を繰り返してもらわない。写真の再投入・再決済を止める。
3. オンボ障害はbioリンクを外す。投稿自体は削除しない。
4. 写真・審査事故は新規承認を止める。
5. 課金事故は`ENABLE_BETA_SUPPORTER_BILLING=false`で課金入口だけを止める。
6. 影響範囲を確認する。DB更新や削除は`PROD-OPERATIONS`の事前確認に従う。
7. 原因コミットが確定している場合だけVercelでロールバックする。
8. 事実と対応をインシデントログへ1行残す。

### P1: 公開は続けて個別対応

- 1人だけの復帰可能な写真読込エラー
- PWA追加案内が出ない、または分かりにくい
- 文字の重なり、余白、写真枠などの表示不良
- 決済後の表示に再読み込みが一度必要
- 説明文やボタンの意味が分かりにくい

同日中に証跡を残す。19:00〜21:00には直さず、翌日以降に判断する。

### P2: 週次へ送る

- 好みの差、文言の磨き、演出の気持ちよさ
- 追加機能の要望
- 一人だけの「こうだったらいい」

返信は受け取りだけにして、実装を約束しない。

### 受け入れを閉じるが障害ではない場合

新規が30人を大きく超える、または19時審査・DM対応を見届けられる人数を超えた場合は、
bioリンクを外す。これはP0ではなく、先行期の品質を守る運用判断である。

## 5. DM返信方針

### 返信時間

- P0疑い: 気づいてから15分以内に受領だけ返す
- はじめられない: 当日2時間以内
- 一般質問・感想: 20:30以降にまとめて、遅くとも翌日
- 即レスを通常運用の約束にしない

### 聞くこと

必要なのは次の4点だけ。

1. iPhone / Android
2. Instagram / LINE / Safari / Chrome / ホーム画面アプリ
3. 止まった画面
4. おおよその時刻

スクリーンショットは可能ならお願いする。カード番号、CVC、写真URL、Storageパス、
認証トークンは送らないよう明記する。DM原文・名前・猫の名前を運用ログへ転載しない。

### 文例1: 使えた・届いた

> 使ってくださってありがとうございます。
> ねこだよりが無事に届いて、ほっとしました。
> また気が向いた日に、一枚のこしてみてください。

質問や感想を追加で求めない。

### 文例2: 写真が読み込めない

> お手数をおかけしてすみません。何度も選び直さなくて大丈夫です。
> iPhoneかAndroidか、どの画面から開いたか（Instagram / LINE / Safari / Chrome）と、
> 止まった画面・おおよその時刻を教えてください。
> エラーが見えていれば、スクリーンショットもお願いします。こちらで確認します。

### 文例3: 最初のねこだよりが来ない

> ごめんなさい。最初の一枚は受け取れているか、こちらで確認します。
> 写真はもう一度入れず、その画面のスクリーンショットと、開いた場所、時刻を教えてください。

### 文例4: 20時を過ぎても届かない

> 待っていてくださったのに、すみません。配達の状態を確認します。
> 写真は入れ直さず、いま見えている画面と、おおよその時刻を教えてください。

開始時刻も確認する。20時前に始めた人は当日20時、20時以降に始めた人は翌日20時が予約時刻である。
全体障害と分かった場合は`OPERATIONS-PLAYBOOK`場面3へ移る。

### 文例5: 写真が二枚ある・別の写真が見える

> ご連絡ありがとうございます。追加の操作はせず、そのままで大丈夫です。
> 表示の状態をこちらで確認します。差し支えなければ、画面全体のスクリーンショットと時刻をお願いします。

他人の情報が見えた疑いがあれば、即P0へ上げる。

### 文例6: 写真は外へ公開されるのか

> 入れた写真がSNSやWebへ公開されることはありません。
> 「届ける」を選んだ一枚だけが、確認のあと、ねてるねこの中で一通として届きます。
> 「自分だけ」を選んだ写真は、ほかの人には届きません。

### 文例7: ホーム画面へ追加できない

> ブラウザのままでも使えます。
> iPhoneはSafariの共有ボタンから「ホーム画面に追加」、AndroidはChromeのメニューから
> 「アプリをインストール」または「ホーム画面に追加」を選べます。
> InstagramやLINEの中では出ないことがあるので、その場合はSafariかChromeで開いてください。

### 文例8: 決済したのに反映されない

> ご連絡ありがとうございます。決済はやり直さず、そのままで大丈夫です。
> 決済したおおよその時刻と、設定画面の表示を教えてください。カード番号などは送らないでください。
> Stripeとアプリの状態をこちらで確認します。

### 文例9: 改善案・要望

> 教えてくださってありがとうございます。いまの使い心地として、受け取りました。
> すぐに変える約束はせず、使われ方を見ながら考えていきます。

### 公開コメントの扱い

- 「使ってみたい」には「プロフィールのリンクから使えます」まで。長い手順を返信しない。
- 不具合の詳細は公開コメントで聞かず、DMへ移す。
- プライバシーの質問には曖昧にせず、「SNSやWebには公開されない」と短く答える。
- 利用人数、残り枠、反応数を返信で誇示しない。
- 批判や好みの違いに説得で返さない。事実誤認だけを短く訂正する。

## 6. 公開前に固定する仮説

人数が10人に届くまでは、割合ではなく「10人中何人」で見る。

| ID | 仮説 | 最初の判定 |
| --- | --- | --- |
| H1 | 最大の自然離脱は入口→写真選択で、写真選択→保存ではない | 写真選択した人の保存失敗は0〜1人。同型2人は技術障害 |
| H2 | 写真を保存できた人には即時便が必ず届く | 保存人数と到着人数の差は即調査。実不成立を確認したらP0 |
| H3 | 即時便は待っている最中の報酬なので、多くがその場で開く | 最初の10人中7人以上が開封。操作不能がなければ未開封は障害にしない |
| H4 | 初回写真の保存と同時に、追加操作なしで次回20時便が予約される | 保存完了者は全員予約済み。既存対象がある場合はそれを維持し、実際の未予約・上書きはP0 |
| H5 | PWA追加は初日完了の条件ではなく、翌日の再訪で効く | 最初の10人中、翌朝までのstandalone起動は2〜4人 |
| H6 | 写真読込のつまずきは通常ブラウザより埋め込みブラウザに偏る | 同じエラーを端末・browser_context別に見る |
| H7 | 自動予約した人のうち、次の20時以降に戻った人には20時便が届く | `evening_delivery_reserved`→`check_started`で再訪を見て、startedに対するsucceededは全員。未再訪は失敗に数えない |
| H8 | 人数が少ない初日は、あめ・むぎや管理在庫がよく届く | 同じ写真の同一人への再配達は0。送り主の偏り自体は許容 |
| H9 | DMはコンセプトへの反論より、開き方・アプリ追加・写真読込に集まる | 型を数えるだけで、個別に感想を聞かない |
| H10 | 自動予約後、次の20時以降に戻る人が最初の継続シグナルになる | 予約→`check_started`→開封を人数で追い、初日だけで成否を決めない |
| H11 | 投稿を見た時に寝顔が手元になく、保存して後から始める人がいる | 最初の1時間だけで需要を判断せず、24時間と翌夜まで見る |
| H12 | 20時前の初回写真は当日、20時以降の初回写真は翌日の20時便へ予約される | 予約日を開始時刻と照合する。境界違い、未予約、既存対象の上書きは不具合として扱う |

βサポーター加入数は投稿3当日の成功指標にしない。日々の交換が無料で成立することを先に見る。

## 7. 観測メモの型

17:15、18:00、19:00、20:15、翌朝に同じ型で記録する。空欄は0とせず「未確認」とする。

```text
## 時刻
入口: N
写真選択: N / 写真保存: N
即時便到着: N / 開封: N / オンボ完了: N
次回20時便予約: N（当日: N / 翌日: N / 既存予約維持: N）
Android: N / iOS: N / 埋め込み: N / 通常ブラウザ: N / standalone: N
要確認エラー: N（型だけ）
未審査: N / 承認: N / 除外: N
DM: N（使えた / 始め方 / 写真 / 配達 / PWA / 課金 / その他）
判断: 継続 / 個別対応 / 入口停止
次に見る時刻:
```

20:15だけ次を追加する。

```text
次回20時便予約: N
20時以降再訪（check started）: N / succeeded: N / failed: N / timeout: N
封筒表示: N / 開封: N
```

## 8. 全体への案内が必要な場合

個別DMで足りる間は全体告知しない。複数人に影響し、入口を閉じた場合だけストーリーズを使う。

### 入口を一時停止

> いま、入口をいったん閉じて確認しています。
> すでにはじめている方で困ったことがあれば、このDMへ教えてください。

### 入口を再開

> 確認ができたので、入口を戻しました。
> 待っていてくださって、ありがとうございました。

### 20時便の遅れ

`OPERATIONS-PLAYBOOK`場面3の文例を使う。技術用語や原因説明は載せない。

## 9. 初日クローズ条件

次を満たせば、初日を「運用継続可」とする。

- 写真保存→即時便到着に欠落がない
- 初回写真保存→次回20時便予約に欠落がなく、同じ予約日の既存対象も維持された
- プライバシー・誤配・共有範囲の事故が0
- 20時便を開始した人は全員成立した
- 未審査を見届けられる件数に収まっている
- P0未解決がない
- DMの型と翌朝に見る項目を記録した

数字が少ないこと、予約済みの人が20時に再訪しないこと、PWAにしないことは初日の失敗としない。
