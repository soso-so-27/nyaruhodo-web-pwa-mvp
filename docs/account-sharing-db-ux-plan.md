# アカウント作成・DB化・家族共有 導入UX設計

## 1. 基本方針

- 初回アクセス時にログインを強制しない
- 「うちの猫、どんなタイプ？」の体験を優先する
- まずは `localStorage` で触れる
- 記録がたまった後に、自然に保存 / 共有 / 引き継ぎへ誘導する
- DB化は技術都合ではなく、ユーザー価値が見えたタイミングで行う

DB化は単なる保存先変更ではなく、ログイン、引き継ぎ、家族共有、課金価値まで含めたUX変更として扱う。

## 2. ログインを出さないタイミング

以下ではログインを出さない。

- 初回アクセス直後
- 猫名入力前
- 3問診断前
- 初回ホーム表示直後
- まだ記録が少ない状態

理由は、初回体験の離脱を防ぐため。最初に感じてほしい価値は、アカウント管理ではなく「この子のことが少し見えてくる」感覚である。

## 3. ログインを出してよいタイミング

以下のタイミングでは自然に出せる。

- 記録が3〜5件たまった時
- 猫を2匹以上追加した時
- 家族共有を押した時
- 病院メモ / ふりかえりを見ようとした時
- 端末引き継ぎをしたい時
- Pro機能を使おうとした時

ログインは「先に必要な手続き」ではなく、「たまった記録をなくさないための選択肢」として出す。

## 4. ログインの見せ方

「ログインしてください」ではなく、ユーザー価値ベースで出す。

候補文言:

- この子の記録を残しておく
- スマホを変えても使えるようにする
- 家族と一緒に見る
- ミケの記録をなくさないようにする
- 家族にもミケの様子を見てもらう

避ける文言:

- アカウント作成
- ログインしてください
- 会員登録
- データ同期設定

## 5. `/cats` の将来役割

`/cats` は以下を置く場所にする。

現在:

- 猫一覧
- いま見ている子
- 猫追加
- 名前変更
- 理解度

将来:

- この子の記録を残しておく
- 家族と共有
- プロフィール
- 写真
- 誕生日
- 性別
- 体重
- かかりつけ病院
- 共有メンバー
- 端末引き継ぎ

ただし、現時点では実装しない。ホームは「今日の猫を見る場所」として保ち、管理や保存に関する導線は必要になった時に `/cats` 側へ置く。

## 6. 家族共有のUX

家族共有はアカウント設定ではなく、猫単位で見せる。

表示例:

- ミケを家族と見る
- ミケの様子を家族に共有

押下後の説明:

> 家族と見るには、まずこの子の記録を保存します。

その後:

- Googleで続ける
- 招待リンクを作る
- 家族を招待する

共有の主語は「アカウント」ではなく「この子」にする。ユーザーにとっての価値は、設定を完了することではなく、猫の様子を家族で見られること。

## 7. DB化の段階

### Phase 1: localStorage MVP

- 今の状態
- 体験検証優先
- `cat_profiles` / `active_cat_id` は端末内保存
- `events` / `diagnoses` / `feedbacks` / `hint_feedbacks` には `local_cat_id` を保存

### Phase 2: 任意ログイン

- 記録を残しておく
- 端末引き継ぎ
- `localStorage` からDBへ移行
- ログインは任意導線として出す

### Phase 3: catsテーブル化

- 猫プロフィールをDB管理
- `active_cat_id` をDB `cat_id` へ移行
- `local_cat_id` は移行期間のみ保持

### Phase 4: 家族共有

- `households`
- `cat_members`
- role管理
- 誰が記録したか

### Phase 5: Pro / 課金

- 家族共有
- 病院メモ
- 長期保存
- ふりかえり
- 通知

## 8. 最小DB構成案

将来の最小追加テーブル候補:

### `profiles`

- `id`
- `display_name`
- `created_at`

### `cats`

- `id`
- `owner_id`
- `name`
- `avatar_url`
- `created_at`

### `cat_members`

- `cat_id`
- `user_id`
- `role`
- `created_at`

### `households`

- `id`
- `name`
- `created_at`

既存テーブルとの関係:

- `events`
- `diagnoses`
- `feedbacks`
- `hint_feedbacks`

移行方針:

- `local_cat_id` は当面残す
- DB `cat_id` へ段階移行
- 既存 `localStorage` 猫はログイン時に引き継ぐ

## 9. localStorageからDBへの移行UX

ログイン後に以下を表示する。

> この端末の猫を引き継ぎますか？

例:

- ミケ
- むぎ

CTA:

- 引き継ぐ
- あとで

注意:

- 勝手に移行しない
- 消える不安を出さない
- 重複猫ができないように注意
- 引き継ぎ後も、ユーザーが「ちゃんと残った」と分かる表示を出す

## 10. 実装開始トリガー

DB化を始める条件:

- 5人以上に継続テストしてもらう
- 複数端末利用の要望が出る
- 家族共有を検証したくなる
- 病院メモ / ふりかえりを作る
- 課金導線を検討する
- ユーザー行動を継続分析したくなる

まだ始めない条件:

- UIが大きく揺れている
- 自分だけで触っている
- 数人に感想だけ聞く段階
- ホーム / 診断体験をまだ調整している
